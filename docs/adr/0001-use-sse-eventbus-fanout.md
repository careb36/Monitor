# ADR 0001 - Use SSE fan-out through `EventBus`

## English

- **Status:** Accepted
- **Date:** 2026-03-28

### Context

The dashboard requires near real-time browser updates for two heterogeneous event sources:

- CDC events consumed from Kafka / Debezium
- infrastructure status changes generated by scheduled polling

Both sources must reach the same frontend with a single browser-friendly transport.
The implementation also needs a low-complexity way to support multiple concurrent dashboard clients.

The current design centers this on:

- `src/main/java/com/monitor/controller/SseController.java`
- `src/main/java/com/monitor/service/EventBus.java`
- `src/main/java/com/monitor/service/KafkaConsumerService.java`
- `src/main/java/com/monitor/service/PollingService.java`
- `frontend/src/hooks/useMonitor.ts`

### Decision

Use Server-Sent Events (SSE) as the browser delivery mechanism, with `EventBus` acting as an in-memory fan-out hub that holds one `SseEmitter` per connected client.

The backend exposes a single stream endpoint at `GET /api/events/stream`, and both producers publish `UnifiedEvent` instances into `EventBus`.
The backend emits event names using `event.getType().name().toLowerCase()`, and the frontend listens explicitly to `data` and `infrastructure` named events.

The stream also sends:

- an initial `:connected` comment on subscription
- periodic `:heartbeat` comments for keep-alive and observability

### Consequences

#### Positive
- a single browser-native streaming mechanism serves the whole dashboard
- producers remain decoupled from HTTP controller code
- the frontend keeps a stable contract based on named SSE events
- initial and periodic heartbeats make connection state observable in development and operations
- multi-client fan-out remains simple and cost-effective for the current deployment model

#### Negative / Trade-offs
- `EventBus` is in-memory and local to one application instance
- horizontal scaling would require a shared event transport or sticky-session strategy
- SSE is one-way; it fits monitoring well but not bidirectional workflows
- connection lifecycle management remains application-side through `SseEmitter`

### Alternatives Considered

- **WebSocket gateway** - more flexible, but unnecessary for one-way operational monitoring
- **HTTP polling from the browser** - simpler infrastructure-wise, but worse latency and more repetitive traffic
- **Separate endpoints per event source** - rejected because the frontend already models a unified monitor state

### Implementation References

- `src/main/java/com/monitor/controller/SseController.java`
- `src/main/java/com/monitor/service/EventBus.java`
- `src/main/java/com/monitor/service/KafkaConsumerService.java`
- `src/main/java/com/monitor/service/PollingService.java`
- `frontend/src/hooks/useMonitor.ts`

## Espanol

- **Estado:** Accepted
- **Fecha:** 2026-03-28

### Contexto

El dashboard necesita actualizaciones en navegador casi en tiempo real para dos fuentes heterogeneas de eventos:

- eventos CDC consumidos desde Kafka / Debezium
- cambios de estado de infraestructura generados por polling programado

Ambas fuentes deben llegar al mismo frontend mediante un unico transporte amigable para navegador.
La implementacion tambien necesita una manera de baja complejidad para soportar multiples clientes concurrentes del dashboard.

El diseno actual se apoya en:

- `src/main/java/com/monitor/controller/SseController.java`
- `src/main/java/com/monitor/service/EventBus.java`
- `src/main/java/com/monitor/service/KafkaConsumerService.java`
- `src/main/java/com/monitor/service/PollingService.java`
- `frontend/src/hooks/useMonitor.ts`

### Decision

Usar Server-Sent Events (SSE) como mecanismo de entrega hacia el navegador, con `EventBus` actuando como hub fan-out en memoria que mantiene un `SseEmitter` por cliente conectado.

El backend expone un unico endpoint de stream en `GET /api/events/stream`, y ambos productores publican instancias `UnifiedEvent` dentro de `EventBus`.
El backend emite nombres de evento usando `event.getType().name().toLowerCase()`, y el frontend escucha explicitamente eventos nombrados `data` e `infrastructure`.

El stream tambien envia:

- un comentario inicial `:connected` al suscribirse
- comentarios periodicos `:heartbeat` para keep-alive y observabilidad

### Consecuencias

#### Positivas
- un unico mecanismo de streaming nativo del navegador sirve a todo el dashboard
- los productores permanecen desacoplados del codigo HTTP del controller
- el frontend mantiene un contrato estable basado en eventos SSE nombrados
- los heartbeats iniciales y periodicos hacen observable el estado de conexion en desarrollo y operacion
- el fan-out multi-cliente sigue siendo simple y costo-efectivo para el modelo de despliegue actual

#### Negativas / Trade-offs
- `EventBus` es en memoria y local a una sola instancia de aplicacion
- el escalado horizontal requeriria un transporte compartido o una estrategia de sticky sessions
- SSE es unidireccional; encaja bien en monitoreo, pero no en flujos bidireccionales
- la gestion del ciclo de vida de conexion sigue estando del lado de la aplicacion mediante `SseEmitter`

### Alternativas Consideradas

- **Gateway WebSocket** - mas flexible, pero innecesario para monitoreo operacional unidireccional
- **HTTP polling desde el navegador** - mas simple desde infraestructura, pero con peor latencia y trafico repetitivo
- **Endpoints separados por fuente de evento** - descartado porque el frontend ya modela un estado unificado de monitoreo

- `src/main/java/com/monitor/service/PollingService.java`

## 🇬🇧 English

### Board-Ready Overview

`Monitor` provides a unified operational view for two classes of risk signals:

- business and platform data anomalies captured through CDC
- infrastructure status changes detected by scheduled health checks

It is designed to improve **time to visibility**, **operator responsiveness**, and **alert resilience** by combining:

- a live monitoring dashboard
- low-latency SSE delivery
- explicit CRITICAL escalation by email

### Solution Summary

At a leadership level, `Monitor` should be understood as a lightweight operational observability layer that:

- consolidates multiple event sources into one operator-facing experience
- reduces delay between detection and visibility
- maintains a resilient alert path by duplicating CRITICAL conditions to both UI and email
- supports local development and controlled delivery through documented workflows and CI gates

### Executive Summary

`Monitor` is a real-time operations monitoring solution designed to unify two operational signals into one dashboard experience:

- **data events** coming from Oracle change data capture (CDC) through Debezium and Kafka
- **infrastructure events** generated by scheduled health checks inside the backend

The platform is optimized for **fast visibility**, **low operational friction**, and **clear escalation paths**. The backend pushes browser updates through **Server-Sent Events (SSE)**, while CRITICAL conditions are intentionally duplicated to **live UI visibility** and **asynchronous email notification**. This gives the project a resilient monitoring posture: operators can react from the dashboard, and on-call engineers can still be notified when they are not watching it live.

### What the system does

- consolidates CDC and infrastructure events into a single event model (`UnifiedEvent`)
- streams events to the frontend through one SSE endpoint: `GET /api/events/stream`
- keeps SSE connections observable through `:connected` and `:heartbeat`
- escalates CRITICAL events to email without blocking the main event pipeline
- separates frontend, backend, and integration responsibilities cleanly

### Technology Stack

| Layer | Technology |
|---|---|
| Backend | Java 21, Spring Boot 4.0.5, Maven |
| Frontend | Next.js 15, React 18, TypeScript |
| Streaming | Server-Sent Events (`SseEmitter`, `EventSource`) |
| Messaging | Kafka + Debezium CDC |
| Alerting | Spring Mail / SMTP |
| Local integration | Docker Compose |

### Information Flow

1. Oracle changes are captured by Debezium and delivered to Kafka.
2. `KafkaConsumerService` parses CDC messages and publishes normalized `UnifiedEvent` instances.
3. `PollingService` emits infrastructure state changes on a schedule.
4. `EventBus` fans out those events to all connected SSE clients.
5. The Next.js dashboard consumes named SSE events (`data`, `infrastructure`) through `useMonitor`.
6. CRITICAL events also trigger `EmailService.sendCriticalAlert(...)` asynchronously.

### Operational Notes

- backend HTTP and SSE run on `http://localhost:8080`
- frontend development runs on `http://localhost:3000`
- local frontend-to-backend API routing depends on the rewrite in `frontend/next.config.js`
- SSE health can be checked with `curl.exe -N --max-time 20 http://localhost:8080/api/events/stream`

### Resumen de la Solucion

`Monitor` es una solución de monitoreo operacional en tiempo real diseñada para unificar dos señales operativas en una sola experiencia de dashboard:

- **eventos de datos** provenientes de change data capture (CDC) de Oracle a través de Debezium y Kafka
- **eventos de infraestructura** generados por health checks programados en el backend

La plataforma está optimizada para **visibilidad rápida**, **baja fricción operativa** y **rutas de escalamiento claras**. El backend empuja actualizaciones al navegador mediante **Server-Sent Events (SSE)**, mientras que las condiciones CRITICAL se duplican intencionalmente hacia **visibilidad en la UI en vivo** y **notificación asíncrona por correo**. Esto le da al proyecto una postura resiliente: los operadores pueden reaccionar desde el dashboard y el personal on-call puede recibir alertas aunque no esté mirando la pantalla en ese momento.

### Qué hace el sistema

- consolida eventos CDC e infraestructura en un modelo único (`UnifiedEvent`)
- transmite eventos al frontend mediante un único endpoint SSE: `GET /api/events/stream`
- mantiene las conexiones SSE observables con `:connected` y `:heartbeat`
- escala eventos CRITICAL a correo sin bloquear el pipeline principal de eventos
- separa claramente responsabilidades de frontend, backend e integraciones

### Flujo de Información

1. Los cambios en Oracle son capturados por Debezium y entregados a Kafka.
2. `KafkaConsumerService` interpreta los mensajes CDC y publica instancias normalizadas de `UnifiedEvent`.
3. `PollingService` emite cambios de estado de infraestructura de forma programada.
4. `EventBus` distribuye esos eventos a todos los clientes SSE conectados.
5. El dashboard Next.js consume eventos SSE nombrados (`data`, `infrastructure`) mediante `useMonitor`.
6. Los eventos CRITICAL también disparan `EmailService.sendCriticalAlert(...)` de manera asíncrona.
